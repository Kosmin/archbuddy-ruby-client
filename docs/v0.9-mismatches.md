# v0.9 implementation — divergences from the design doc

Ground-truth design: `.context/plans/v0.9-QUEUED-from-cache-graph.md` (in the
kosmin-claude-skills workspace). Implemented on `feat/v0.9-report-graph`.
This file records where the implementation diverged from the design sketch /
task framing, and why.

## D1 — "per-class files carry scores/clutter" is NOT the committed contract

The task framing said `context.ranked` should be "a bottleneck list from the
committed **per-class scores**" and that "per-class files carry scores/clutter".

**Reality (verified against `Cache::Writer#build_fragment` + a live committed
tree):** the committed detail-tree fragments carry per-node STRUCTURAL fields
only — `symbol`, `kind`, `class`, `branches`, `decisions`, `entrypoint` — and
per-file `edges` (`from`/`to`/`calls`). They do NOT carry `clutter_score` or the
8 per-node metrics. Those live in the OPAQUE `findings.yml` (keyed by opaque id),
which the committed layer intentionally does not inline (R11).

The only committed, per-symbol, real-name CLUTTER signal is the root aggregate's
`multiplexer_proxies` list (`{symbol, added_coupling}`, worst-first, VERBATIM
from the engine).

**Decision:** `context.ranked` is built from the committed `multiplexer_proxies`
(`clutter_score = added_coupling`). This is the faithful, contract-accurate
"real committed clutter" ranking the design wanted ("top-N real hotspots"),
without inventing a per-node score the committed cache does not hold. Graph nodes
that are not proxies simply carry no clutter and rank last in the `--max-nodes`
cap — matching the html_formatter's existing `Float::INFINITY` tiebreak.

## D2 — `report` target-path arg (W1)

W1 listed "`report` in `<target>`". `report` has no `Dir.pwd`-based Writer/Checker
(the write-target bug); it only READS the committed aggregate via a repo-relative
path, which already works when run from inside the target ("report in <target>").
`report`'s only positional arg is the LEGACY `FINDINGS_YML` path, so adding a new
target positional would collide with it. **Decision:** left `report`'s read path
as repo-relative (run from the target dir); the write-target fix (the actual bug)
is applied to collect/analyze/reset/`--check`.

## D3 — no CHANGELOG.md in the client repo

W3 said to update CHANGELOG if it describes the from_cache graph limitation. The
client repo has no CHANGELOG.md. **Decision:** updated `README.md` +
`ARCHITECTURE.md` (which DID describe the limitation) instead; no CHANGELOG created.
